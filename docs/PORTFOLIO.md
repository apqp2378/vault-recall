# 포트폴리오 서사 (PAR) — vault-recall

**P(문제):** 지식을 GitHub star·노트로 저장해도 결정 순간(면접·실무)에 소환되지 않는다.
이전 두 프로젝트가 이 문제의 반쪽씩을 다뤘지만 — topic-shelf는 수집에서, knowledge-ops는
일회성 진단에서 멈췄고, '소환'은 키워드 매칭 데모 수준이었다.

**A(분석·실행):** 두 레포를 하나의 루프로 재설계했다.
수집(ingest) → 구조화(Obsidian 파서·실링크 그래프) → 소환(BM25+그래프 부스트 하이브리드,
근거 인용·정직한 공백) → 진단(orphan·검증률·연결 우선순위) → 증빙(골드셋 Recall@k·MRR).
코어는 의존성 0의 결정적 계산으로 유지하고, 임베딩·LLM은 provider 확장점으로 분리했다.

**R(결과):** 실제 개인 지식볼트(188 노트)에서 재현 가능한 수치로 검증 —
(README의 실측 수치 참조). 테스트 12건, 골드셋 평가 내장, 단일 CLI로 스캔→소환→진단→평가.

**면접 꼬리질문 대비:**
- 왜 BM25인가? — 임베딩 없이도 재현 가능한 기준선. 한국어는 음절 bigram으로 대응, 임베딩은 RRF 융합 확장점.
- 그래프 부스트의 근거는? — 위키링크는 사람이 승인한 관계라서 '함께 소환'의 신호로 신뢰 가능.
- 할루시네이션은? — 생성을 안 한다. 소환+인용만. 근거 부족이면 공백 선언(테스트로 고정).
